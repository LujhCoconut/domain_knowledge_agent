# CoreSec (OSDI'26)

- **来源**: OSDI '26 (Operational Systems Track), https://www.usenix.org/system/files/osdi26-gaikwad.pdf
- **全称**: The Abstention Protocol: RCA for Clos Fabrics
- **系统名**: CoreSec
- **作者**: Madhava Gaikwad (Microsoft), Deepak Pandey (Microsoft)
- **类型**: 论文-系统 (Operational Systems — network operations / RCA)
- **一句话 TL;DR**: 大规模 Clos 网络中的根因分析 (RCA) 面临**嘈杂、部分、异步遥测**的挑战。传统加权打分方法在这些条件下不稳定。CoreSec 用 **PAM 风格的弃权代数**替代加权融合——遥测代理通过控制标志组合，产生确定性决策，并在证据不足时**显式弃权**。结合拓扑感知的故障面配置，CoreSec 在超大规模部署中提供稳定、可解释的 RCA，无需重调参数。

## 重要术语解释

| 术语 | 解释 | 在本文中的作用 |
|------|------|---------------|
| **RCA** (Root Cause Analysis) | 从大量遥测信号中定位故障根本原因 | CoreSec 的核心任务 |
| **Clos fabric** | 三级交换结构（Server→TOR→T1→T2 Spine），提供任意两端点间的多等价路径 | 被分析的目标网络拓扑 |
| **Gray failures** | 非 fail-stop 的故障：部分丢包、间歇性抖动、概率性错误 | 使 RCA 极其困难的主要原因 |
| **PAM** (Pluggable Authentication Module) | Unix 认证的标准链式组合框架（success/ignore/abstain/deny） | CoreSec 弃权代数的灵感来源 |
| **Abstention algebra** | 显式"弃权"作为一种一等决策结果（而非强行给出确定性答案） | CoreSec 的核心创新 |
| **Control flags** | 类似于 PAM 的 success/ignore/abstain/deny 决策语义 | 组合多个遥测代理信号的方式 |
| **Score-based RCA** | 对多个遥测信号加权求和，最高分=根因 | 传统方法——在嘈杂环境下不稳定 |
| **Monotonic convergence** | 随着更多证据累积，RCA 决策稳步向正确根因收敛（不会来回翻转） | CoreSec 的关键理论性质 |

## 背景与动机

### 问题
- 大规模云网络运行在**持续背景故障**中（CRC 错误线缆、链路抖动、交换机升级）
- Clos 拓扑通过多路径**吸收**这些故障，但也使 RCA 极其困难
- **灰色故障**是常态而非异常：仅 2% 丢包、内存位翻转仅影响特定 ECMP 路径、固件 bug 间歇性重启交换机
- 当客户工作负载故障时，**数十个实体同时显示异常**——大多数只是常规的背景噪声

### 为什么传统打分方法失败

加权求和多个遥测信号的分数，然后选最高分作为根因，在以下情况下不稳定：
- 遥测是**嘈杂的**（正常信号看起来像异常）
- 遥测是**部分的**（某些数据源缺失或延迟）
- 遥测是**异步的**（不同来源的时间戳未对齐）
- 一个小权重变化可能导致根因从交换机 A 翻转到交换机 B → 决策不稳定

### 核心洞察
> 当证据模糊时，显式说"我不知道"（弃权）比强行给出可能错误的确定性答案更好。

CoreSec 从 PAM (Unix 可插拔认证模块) 借鉴设计灵感：PAM 的 success/ignore/abstain/deny 控制标志组合语义在安全领域已被证明是稳健的。将相同的思想移植到网络 RCA：远程代理是"遥测代理"，控制标志定义它们如何共同决策。

## 方案介绍

### 弃权代数 (§3-4)

- 借鉴 PAM：success = 此代理找到匹配的根因；ignore = 不相关；abstain = 证据模糊，拒绝判断
- **显式弃权**作为一等结果：当两个代理冲突或单个代理无法确定时 → abstain → 等待更多证据
- **确定性决策**：相同的输入总是产生相同的决策（无概率性翻转）

### 拓扑感知配置 (§5)

- Clos fabric 物理结构编码为**故障面**：每个链路/交换机的故障影响可预测的端点集合
- 遥测代理仅需要在给定故障面的范围内查找信号
- **单调收敛**：随着更多遥测数据到达，CoreSec 的决策稳步向正确根因收敛，从不翻转

### 生产部署架构

- 部署在 Microsoft Azure 的超大规模 Clos 网络中
- 无需重调参数（跨部署环境的稳定行为）
- 工程团队使用聚合 RCA 数据识别固件/光模块/运维流程中的系统性弱点

## 整体评估

### 真正的新意
1. **将 PAM 弃权模式移植到 RCA**：从操作系统安全认证框架到网络根因分析的跨领域移植
2. **"弃权作为一等决策"**：在嘈杂、不完整的遥测下，显式承认不确定性优于强制给出可能错误的答案
3. **单调收敛作为正确性准则**：随着更多证据累积，决策持续接近真相，不会"来回翻转"

### 优点
- **生产部署和验证**：Microsoft Azure 超大规模网络的真实 RCA
- **可解释性**：与"黑盒加权分数"不同，弃权代数产生可理解的决策路径
- **不需要重调参数**：在不同环境下行为稳定
- **处理灰色故障的明确策略**：大多数生产故障不是 fail-stop——CoreSec 明确为此设计

### 局限
- **仅 Clos 拓扑**：拓扑感知配置依赖 Clos 的特定结构——向其他网络拓扑的推广可能需要工作
- **PAM 代数限制**：组合语义是成功的但对于某些场景可能不够表达（例如需要概率聚合而非确定性组合）
- **论文未披露定量指标**：未提供与打分系统相比的具体准确性/召回率数字（可能因生产数据敏感性）

### 可复用启发

1. **"弃权"作为系统设计中的一等决策状态**：不仅适用于 RCA——任何需要从嘈杂/部分数据中做决策的系统都可以受益于显式的不确定性表示
2. **PAM 组合语义是通用的**：success/ignore/abstain/deny 模式可以推广到任何需要组合多个独立判断的场景（异常检测、故障定位、策略执行）
3. **拓扑感知 + 弃权**的组合：将可预测的物理拓扑与显式的不确定性处理相结合，是处理大规模物理基础设施的强模式
4. **"单调收敛"作为在线系统的正确性准则**：比"最终一致"更强——决策质量随时间单调提升，从不退化
