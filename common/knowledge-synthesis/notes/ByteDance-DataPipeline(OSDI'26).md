# ByteDance Data Pipeline (OSDI'26)

- **来源**: OSDI '26 (Operational Systems Track), https://www.usenix.org/system/files/osdi26-chen-luofan.pdf
- **全称**: Teaching The Old Dog New Tricks: Building Efficient Data Pipelines for Large-Scale LLM Pre-training
- **作者**: Luofan Chen*, Chenhan Wang* (USTC & ByteDance Seed), Weidong Zhang, Jinxin Chi, Hequan Zhang 等 (ByteDance), Cheng Li (USTC & Hefei Comprehensive National Science Center)
- **类型**: 论文-系统 (Operational Systems — data infrastructure)
- **一句话 TL;DR**: 对 30,000 个训练作业 90 天生产 trace 的分析揭示 LLM 预训练数据管线的三个被忽视的瓶颈：跨数据中心评估延迟、启动时 checkpoint 加载 I/O 竞争、多模态数据转换的 CPU 瓶颈。三个非系统特定的优化——全局命名空间预测 checkpoint 复制、主动热文件复制、将数据转换卸载到存储层 CPU——将每次评估浪费的 GPU 小时从 **16,800 降至 4,000**，checkpoint 加载时间缩短 **40.8%**，数据加载导致的训练停顿减少 **63.2%**。

## 重要术语解释

| 术语 | 解释 | 在本文中的作用 |
|------|------|---------------|
| **Cross-DC Latency Trap** | 跨数据中心获取远程 checkpoint 进行伴随评估的延迟 | 瓶颈 #1 |
| **Companion evaluation** | 与训练并行的 out-of-band 验证管线——加载最近的 checkpoint 并运行基准测试 | 训练质量标准保障 |
| **HDFS** | Hadoop 分布式文件系统 | ByteDance 的训练数据存储后端（EB 级） |
| **Checkpoint loading I/O contention** | 作业启动时数千 GPU 同时读取 checkpoint → 存储层热点 | 瓶颈 #2 |
| **Data transformation bottleneck** | 多模态模型的 CPU 密集型数据预处理（解码/调整大小/增强）在训练 GPU 节点上耗尽 CPU | 瓶颈 #3 |
| **Global namespace predictive replication** | 利用全局命名空间跨数据中心的可预测性先行复制 checkpoint | 优化 #1 |
| **Proactive hot-file replication** | 识别常见 checkpoint 文件的热读模式 → 在故障前主动创建副本 | 优化 #2 |
| **Storage-tier CPU offloading** | 将数据转换推送到存储节点自身的 CPU（HDFS DataNode），而非训练节点 | 优化 #3 |

## 背景与动机

### 问题
- 数据管线已成为 LLM 预训练的首要性能决定因素，但受到的审视（相较于 MFU）要少得多
- ByteDance 管理着跨越**数十亿到万亿参数**模型的任务，使用 HDFS 作为 EB 级存储骨干
- 三个被忽视的瓶颈通过对 30,000 个训练作业的 90 天 trace 分析得以识别

### 三个瓶颈

**瓶颈 1：跨数据中心延迟陷阱**
- 伴随评估每隔 N 步加载 checkpoint 并从远程数据中心运行验证
- 评估作业对硬件的要求与训练不同 → 通常在其他数据中心运行
- 跨 DC 获取 checkpoint 的数据传输主导了延迟 → **训练 GPU 在等待评估结果时处于空闲状态**
- 生产数据：每次评估浪费 ~16,800 GPU 小时

**瓶颈 2：启动阶段的 Checkpoint 加载 I/O 竞争**
- 当训练作业启动时，数千 GPU 同时发起 checkpoint 读取
- HDFS 存储节点上的"闪电群"效应 → 热点和 I/O 竞争
- 延迟了作业初始化 → 启动时间增加 ~40%

**瓶颈 3：多模态数据转换的 CPU 瓶颈**
- 图像/视频训练数据需要在加载过程中进行解码、调整大小和增强
- 这些是 CPU 密集型操作 → 使训练节点上的 CPU 饱和→ 使 GPU 停滞
- 对于多模态模型，这是数据加载最主要的开销

### 为什么是"旧把戏"

论文标题暗示了核心哲学：这些优化**不是系统特定的**（不需要专门的硬件或新的存储系统），而是通过解决基础架构不匹配问题，为现有基础设施（HDFS 等"老狗"）带来高回报：
- 预测复制 → 利用跨 DC 的全局命名空间的可预测性
- 热文件复制 → 利用 HDFS 现有的复制机制，但以主动、预测的方式
- 存储层 CPU 卸载 → 利用 HDFS DataNode 上已有的 CPU 资源（而非需要新的计算集群）

### 我的分析
这是 OSDI '26 中的第三篇 Operational Systems 论文（继 USEC 和 Tessera 之后），也是**唯一关于数据基础设施**的论文。它与其他训练论文（Tessera、Hetu-v2、Syncopate）不同——那些解决的是计算/通信瓶颈，而这一篇解决的是**数据供给瓶颈**——即在庞大训练作业背景下"如何可靠高效地移动 EB 级数据"。最引人注目的统计数据是"16,800 GPU 小时每次评估被浪费"——这相当于空闲的计算资源因糟糕的数据管线而被浪费。

## 方案介绍

### 三个优化

**优化 1：基于全局命名空间的预测 Checkpoint 复制**
- 观察：跨 DC 的 checkpoint 访问模式遵循**可预测的命名空间模式**（哪个 DC 的哪个集群运行哪个评估作业）
- 方法：分析全局命名空间 → 预测哪些 checkpoint 将被远程 DC 访问 → 在评估作业需要它们之前主动复制
- 效果：将每次评估浪费的 GPU 小时从 **16,800 降至 4,000**（减少了 **76%** 的浪费）

**优化 2：主动热文件复制**
- 观察：作业启动期间的 checkpoint 加载展示了**清晰的"热点"模式**——某些文件/块被大量并发读取
- 方法：识别这些模式 → 使用 HDFS 现有的复制原语，主动在**故障前**创建额外的副本 → 分散读取负载，减少 I/O 竞争
- 效果：将 checkpoint 加载时间缩短 **40.8%**

**优化 3：将数据转换卸载到存储层 CPU**
- 观察：多模态训练的数据预处理（JPEG 解码、视频帧提取、张量变换）是 CPU 密集型的，会饱和训练节点 CPU
- 方法：将转换逻辑推向**存储节点**（HDFS DataNode）→ 在每个块内进行就地转换 → 训练节点接收"即插即用"数据
- 效果：数据加载导致的训练停顿减少 **63.2%**

### 非系统特定的设计
- 所有三个优化均使用**现有的基础存储原语**（HDFS 复制、命名空间元数据、DataNode 计算）
- 无需专用集群、专用硬件或新的存储格式
- 对类似使用通用存储系统进行 GPU 训练的任何组织均适用

## 证据与评估

### 测试环境
- 字节跳动的生产预训练集群：数千 GPU、EB 级 HDFS 存储
- 90 天内收集的 30,000 个训练作业 trace
- 跨文本和多模态模型（从十亿参数到万亿参数规模）

### 关键结果

| 指标 | 改进 | 机制 |
|------|------|------|
| 每次评估浪费的 GPU 小时 | **16,800 → 4,000**（减少 76%） | 预测 checkpoint 复制 |
| 启动时 Checkpoint 加载时间 | **减少 40.8%** | 主动热文件复制 |
| 数据加载导致的训练停顿 | **减少 63.2%** | 存储层数据转换卸载 |

## 整体评估

### 真正的新意
1. **首次量化大规模训练中数据管线瓶颈的系统性研究**：之前的工作关注 MFU（计算）和通信重叠（网络），但数据供给受到的系统性分析要少得多
2. **"利用现有基础设施"的设计理念**：不提议取代 HDFS 或部署新的存储系统——而是通过预测复制、主动热点缓解和存储层计算在**现有基础之上**工作
3. **全局命名空间分析作为预测信号**：跨 DC checkpoint 访问的可预测性被用作主动复制的信号——将被动基础存储操作转变为主动优化

### 优点
- **生产规模 trace**：30,000 个作业 / 90 天 → 高度可信
- **非系统特定**：适用于任何 HDFS 部署（且对其他分布式存储系统也有概念上的可移植性）
- **解决了一个被忽视的领域**：数据管线在训练优化中通常被低估，但却是首要的 GPU 空闲原因
- **全面的覆盖范围**：跨 DC（广域网）、启动（I/O 竞争）和稳态（CPU 密集型转换）——三个生命周期阶段

### 局限
- **HDFS 特定**：优化利用 HDFS 的特定原语（DataNode 计算、块复制策略）——向其他存储系统的移植需要适配
- **预测复制假设模式稳定**：如果在作业执行期间跨 DC 的评估模式发生巨大变化，预测检查点复制可能会失效
- **存储层 CPU 卸载在计算压力大的存储节点上可能受限**：如果 DataNode 已经饱和，增加的转换工作可能竞争或降低存储吞吐量

### 可复用启发

1. **"数据供给"应被视为第一等的训练效率瓶颈**——与 MFU 和通信重叠并列。GPU 空闲不应仅仅归因于计算，还应包括"数据没有按时到达"
2. **"老基础设施 + 预测优化"是比"部署新系统"成本更低的基础设施升级路径**
3. **全局命名空间 + 访问模式分析 → 主动数据放置**：适用于任何跨数据中心的数据密集型工作流
4. **"将计算带到数据"原则在 AI 训练中的体现**：将转换卸载到存储层，而非传输原始数据并在训练节点转换
