# Cocoon(OSDI'26)

- **来源**: OSDI '26, https://www.usenix.org/system/files/osdi26-kim-donghwan.pdf
- **全称**: Cocoon: A System Architecture for Differentially Private Training with Correlated Noises
- **作者**: Donghwan Kim, Xin Gu (Penn State), Jinho Baek, Timothy Lo, Younghoon Min, Kwangsik Shin, Jongryool Kim (SK hynix), Jongse Park (KAIST), Kiwan Maeng (Penn State)
- **类型**: 论文-系统 (privacy + systems + near-memory processing)
- **一句话 TL;DR**: DP 训练用**相关噪声**（跨迭代抵消以提升准确率）正在被 Google/Apple/Microsoft 广泛采用，但其系统开销从未被系统研究过。Cocoon 首次量化了相关噪声的存储/计算瓶颈：**噪声历史内存可能超过参数内存本身**（尤其在 embedding table 场景）。Cocoon 通过 CPU-GPU-NMP 三级噪声分布存储 + 稀疏 embedding 优化 + FPGA NMP 加速，实现 **1.23-10.82×** 加速。

## 重要术语解释

| 术语 | 解释 | 在本文中的作用 |
|------|------|---------------|
| **DP** (Differential Privacy) | 给梯度添加噪声以提供隐私保证的数学框架 | 根本问题域 |
| **Correlated noise** | 跨迭代相关的噪声——后期噪声可部分抵消前期噪声 → 比独立高斯噪声更少的准确率下降 | Cocoon 优化的目标机制 |
| **Noise history** | 早期迭代的噪声值存储，后期需要用于抵消计算 | 主要的系统瓶颈：内存占用可能超过参数内存 |
| **Mixing matrix (C)** | 跨迭代噪声的线性变换矩阵（GEMV 操作） | 每步训练的核心计算开销 |
| **NMP** (Near-Memory Processing) | 在内存设备内部进行计算（而非将数据移动到 CPU/GPU） | Cocoon 的加速硬件 |
| **Embedding table** | 稀疏特征的大查找表（用于推荐模型） | 相关噪声开销特别严重的场景 |
| **FPGA NMP prototype** | SK hynix 的 FPGA 原型，在内存设备内部加速矩阵向量乘法和噪声管理 | 真实的硬件原型 |

## 背景与动机

### 问题
- DP 训练中添加的噪声随训练累积 → 显著降低模型准确率 → **这是 DP 采用的主要障碍**
- **Correlated noise** 通过使跨迭代的噪声**部分抵消**来解决准确率问题，已经被 Google Gboard、Apple、Microsoft SwiftKey 部署
- 然而，相关噪声相关工作的焦点一直是**准确率**提升，集中在资源充裕的场景（例如，用 1024 台机器训练一个十亿参数模型）——而**系统效率**从未被研究

### 相关噪声的系统瓶颈

生成相关噪声涉及：
1. **存储整个迭代的噪声历史**（GB 级）——用于后期抵消
2. **每步训练计算矩阵向量乘法**（`C × history`）——生成新噪声
3. 对于**大型嵌入表**（推荐模型），每个参数的噪声状态扩展为嵌入表维度 × 步数 → **噪声历史内存可能超过参数内存本身**

### 表征研究
Cocoon 进行了首次系统表征，揭示：
- 当模型相对硬件容量较大时，相关噪声引入不可忽略的开销
- 大型嵌入表使问题恶化——每个稀疏特征的噪声历史独立存储
- 现有系统完全忽略了这些开销

### 我的分析
这是 OSDI '26 的第二篇隐私论文（继 Ote 之后），也是第一篇将 DP 训练作为系统问题处理的论文。前面的 24 篇论文主要关注性能（CXL/LLM 推理/训练），而 Cocoon 将隐私作为一等系统约束引入。最有趣的方面是：相关噪声是社区**为了准确率**而采用的——他们乐于在准确性上获得回报，但从未测量系统成本。Cocoon 首次揭示了"存储噪声历史"和"矩阵向量乘法"是主要瓶颈。

## 方案介绍

### 三级噪声历史存储层级

```
GPU HBM: 当前迭代的活跃噪声（热）
CPU DRAM: 近期历史（温）
Memory extension module (NMP): 长期历史（冷）
```

- 将大噪声历史**切分**到 GPU/CPU/内存扩展模块
- 在三级层次结构中**并发处理**

### 稀疏嵌入表优化

- 嵌入表在每个参数（每个特征 × 嵌入维度）有独立的噪声状态
- 观察：每次迭代仅访问嵌入表的**少量行**（稀疏访问）
- Cocoon：预取近期行、惰性驱逐冷行、为嵌入表的噪声存储使用**基于行的格式**

### NMP 加速

- 目标：将噪声历史的矩阵向量乘法（GEMV）卸载到**FPGA NMP 设备**
- 使用 SK hynix 的 FPGA 原型（即将商业化的 NMP 设备）
- NMP 在数据驻留的地方执行 GEMV → 消除了将整个噪声历史传输到 GPU 的需求

## 证据与评估

### 测试环境
- 真实系统 + FPGA NMP 原型（SK hynix）
- DP 训练的代表性模型和数据集（文本、推荐）
- 对比：没有相关噪声优化的 GPU-only baseline

### 关键结果

| 指标 | 结果 | 说明 |
|------|------|------|
| 端到端加速 | **1.23-10.82×** | 跨模型大小和工作负载的变体 |
| 内存 footprint | 显著减少 | 噪声历史在 CPU/GPU/NMP 之间分布 |
| NMP 加速 | 矩阵向量乘法瓶颈被 eliminate | FPGA 处理相关噪声计算 |

## 整体评估

### 真正的新意
1. **首次对相关噪声进行系统表征**：揭示了被忽视的内存和计算开销——之前的所有工作都只关注准确率
2. **用于 DP 噪声的 CPU-GPU-NMP 内存层级**：类似常规数据的分层缓存，但应用于噪声历史——这是一个新颖的应用
3. **将 NMP 用于隐私计算**：NMP 大多用于图/数据库工作负载——Cocoon 证明它对于 DP 训练的噪声管理同样有价值

### 优点
- **及时**：相关噪声正在被 Google/Apple/Microsoft 实际部署——系统支持是迫切需要的
- **真实的硬件原型**：SK hynix FPGA NMP，而非纯仿真
- **综合的瓶颈表征**：在确定优化方案之前展示了"什么导致相关噪声变慢"
- **涵盖两种模型类型**：密集 Transformer 和稀疏嵌入表——广泛的适用性

### 局限
- **NMP 硬件可用性**：FPGA 原型不是量产硬件——最终部署取决于 SK hynix NMP 的商业化
- **仅相关噪声**：不针对所有 DP 机制——特定于"带抵消的相关噪声"这一家族
- **嵌入表优化是启发式的**：基于行的预取/驱逐——不能保证所有访问模式下都最优

### 可复用启发

1. **"为准确率优化而添加的特性"需要系统表征**：社区为准确率而采用相关噪声，对系统成本的测量为 0——这对于任何为准确率引入复杂性的新特性（量化、稀疏化、混合精度）都是常见的模式
2. **"噪声即数据"——需要分层管理**：噪声历史本质上就是一个新的、大型的、迭代访问的"数据"维度——应该享受与训练数据本身相同的分层存储和管理
3. **NMP 可以在隐私计算中发挥关键作用**：隐私工作负载充斥着"大数据、简单计算（GEMV）"的模式——与 NMP 的优势完美契合
